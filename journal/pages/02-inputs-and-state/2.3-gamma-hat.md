# γ̂ as a Stage-4 lowering choice

The one-body density matrix `γ̂` (one of the seven state DOFs in
`arch-04-state`) is the most demanding object in the state vector: a
single logical entity with multiple inequivalent encodings, where
different operations have different runtime cost on different
encodings.

Under the `PhysicsGraph` framing (`arch-06-physics-graph`) and the
compose-time pipeline (`arch-07-pipeline`), `γ̂` is one node-type
whose Stage-4 `CompressionPlan` ranges over a structured `Basis ×
Form` product. This file documents that product, the scorer-only boundary
that bounds it (§15.2), and how the four questions once listed as open
data-structure problems resolved (§15.4).

## 15.1 The encoding vocabulary

An encoding factors into two orthogonal axes:

```
Basis ∈ { Real, Reciprocal, Wannier, NaturalOrbital, SymmetryAdapted }
Form  ∈ { Dense, Sparse, BlockDiag, LowRank }
```

First-class V1 pairs:

- `(Reciprocal, BlockDiag)` for periodic substrates (the diamond MVP
  default).
- `(Real, Sparse)` for defects, surfaces, amorphous regions.
- `(Wannier, Sparse)` for interface layers and dangling bonds.
- `(NaturalOrbital, LowRank)` for low-rank substrates.
- `(SymmetryAdapted, BlockDiag)` for the output of
  `SymmetryAdaptedHamiltonianOf` (`arch-09-vocabularies §9.2`).

Stage 4 selects one slot per density-matrix-typed node based on the
`(PeriodicityStructure, SiteDecoration)` tuple. Transcoders convert
on demand for operations whose runtime cost is lower in a different
encoding.

## 15.2 Read path vs write path

**`/physics` is scorer-only: `γ̂` never evolves inside the oracle.**
States arrive complete, are scored, and are discarded; there is no
trajectory in here and therefore no `γ̂` with a history. This is not new —
`[arch-02-libraries]` and the README already say `/physics` "does not hold
time-varying state values … integrate trajectories". This section used to
list *time-stepping* as a `γ̂` write path, which contradicted them; canon
wins and the write path is construction and self-consistency only.

An EOM-violation residual over `γ̂` (`arch-11-residuals`) is consistent with
this and is not an exception: it *scores* a supplied `∂_t γ̂` against
`L·δE/δx + M·δS/δx`. Scoring a proposed rate is not taking a step.

Most `γ̂` traffic during PINO training is **reads** — apply `Ĥ`,
extract density, take a trace, take an eigendecomposition. A small
minority is **writes** — construction and the self-consistent step. The
two paths differ:

```
READ PATH (dominates evaluation):
    interface ──destructor──▶ tensor-network substrate
    (lazy materialization; no term staging, no bundle sync)

WRITE PATH (construction, self-consistent step):
    interface ▶ term-algebra ▶ planner ▶ (encoding) ▶ substrate
```

Under the always-cheap pipeline, the read path is what the runtime
kernel does; the write path is absorbed by Stage 4 codegen.

**Where trajectory ownership went.** `/physics` does emit a per-tier
tangent map and a steppable-form manifest (`[arch-18-open-decisions]`,
integrator interface) — a pure function, not an integrator. A consumer
that integrates the micro tier owns `γ̂`'s representation health across its
own steps, and the manifest declares what it needs to do that. The drift
problem is **exported, not dissolved**, and saying so is the point.

Self-consistency (when `Ĥ[γ̂]` depends on `γ̂`) is *structured* by the
coalgebraic fixed-point form but **solved by the implicit-diff adjoint**
synthesized at Stage 4 (`arch-07-pipeline §7.4`); convergence work
happens via explicit iteration above the substrate.

## 15.3 Strategies as Stage-4 internals

Several representation strategies that might naively look like
architectural peers are, under the always-cheap framing, Stage-4
codegen tactics applied to nodes whose `type` is the density-matrix
typeclass:

| Strategy | Realization |
|---|---|
| Codata / coalgebraic interface | The `Node` interface itself (`arch-06-physics-graph §6.1`); destructors are method invocations |
| Typed term algebra (staging) | Stage 1 / Stage 3 internal — the symbolic IR for compose-time |
| E-graph with equality saturation | Optional offline rewrite oracle; not on the runtime path |
| Pullback bundle of synchronized encodings | Single-slot V1 (one canonical encoding per node); the bundle is a multi-slot V2 generalization |
| Tensor network with cost-aware contraction | Stage-4 codegen primitive for `Form ∈ {LowRank, BlockDiag}` |

## 15.4 The four questions, and where each went

This section listed four "genuinely open CS problems". They were one
problem — exact content-addressed identity meeting approximate numerics —
seen from four sides, and the resolution is `arch-20-representations
§20.4.1`/`§20.4.2`: **identity stays exact, ε is estimated beside it.**
Each of the four resolves under that rule plus the scorer-only decision
(§15.2). None is still open here.

- **Approximate vs exact equality (ε-equivalence)** → **resolved, and it
  was the keystone.** The node identity contract is *not*
  bisimulation-up-to-ε and must not be: `≈_ε` is not transitive, so it
  yields no quotient, no canonical representative and nothing to hash
  (`§20.4.1`). Identity is exact; a rewrite is admitted when it is exact
  over ℝ *and* its float side conditions are discharged by an e-class
  analysis carrying interval and not-equals facts — the mechanism that
  made Herbie's rule set sound (`arch-07-pipeline §7.3`). E-graphs stay
  offline, but now for a stated reason rather than as a hedge against an
  unsolved problem.

- **Materialization policy** → **misfiled, not open.** It is not an ε
  problem: forcing versus deferring materialization changes *cost*, not
  *value*, so there is no error term at all. It is the classic
  rematerialization-versus-storage trade-off, which `revolve`
  (Griewank & Walther, *ACM TOMS* 26(1) 19–45, 2000) solves optimally for a
  chain and which is **NP-complete** on a general DAG (Naumann, *J. Discrete
  Algorithms* 7(4) 402–410, 2009 — a separate result, not `revolve`'s). It belongs to Stage-4 codegen as a schedule over the
  adjoint tape (`arch-07-pipeline §7.4`), not to this page.

- **Long-trajectory drift and rank growth** → **not this library's
  problem, and declared as such.** Under §15.2 there is no trajectory
  inside `/physics`, so nothing accumulates. A consumer that integrates
  the micro tier does face it, and the literature has an answer: the
  robust dynamical-low-rank integrators — projector-splitting (Lubich &
  Oseledets, *BIT* 54, 2014) and basis-update & Galerkin (Ceruti &
  Lubich, *BIT* 62(1) 23–44, 2022) — whose error bounds are
  **independent of small singular values** (Kieri, Lubich & Walach,
  *SIAM J. Numer. Anal.* 54, 2016 for the former; Ceruti & Lubich for
  the latter, which is the paper that carries the property across),
  unlike every standard integrator. The rank-adaptive BUG integrator (Ceruti, Kusch & Lubich,
  *BIT* 62, 2022) truncates to a declared tolerance and, up to it,
  preserves the norm when the equation does, the energy for Hamiltonian
  systems, and monotone decrease in gradient flows. Those three map
  term-for-term onto trace, the `L` block and the `M` block of
  `arch-05-generic`. The manifest carries the hand-off
  (`[arch-18-open-decisions]`).

- **Rank-dependent applicability** → **a compile-time predicate.** With no
  accumulation to monitor, "is the `(NaturalOrbital, LowRank)` slot
  applicable?" is decided at Stage 4 from `(PeriodicityStructure,
  SiteDecoration)`, alongside every other slot choice in §15.1 — not by a
  runtime check whose cost was the original objection. Four-index objects
  (BSE, BTE collision matrices) go to TT compression directly
  (`arch-07-pipeline §7.4`), as before; what changes is that this is a
  selection rule rather than an open question.

## 15.5 Diamond MVP commitment

The diamond MVP runs on `(Reciprocal, BlockDiag)` with each k-block
stored as orbitals (low-rank in the band index). At MVP scale this
sizes at ~18 MB total; densifying would cost ~460 MB at the same mesh,
which is exactly why the encoding forbids it (`mvp-02-gamma-budget`).
